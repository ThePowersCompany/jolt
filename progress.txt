2026-05-11 PRD #9 — Unified error type: JoltRError trait with Into<Response>

Completed:
- Added joltr-core/src/error.rs with `JoltRError` trait (status(), message(), default to_response()) and `ErrorBody { error: String, status: u16 }` JSON body (impl JsonBody).
- Registered module in joltr-core/src/lib.rs and re-exported `ErrorBody`, `JoltRError` at crate root.
- Added joltr-core/tests/error.rs with 4 tests: status/body field mapping, axum JSON bridge + Content-Type, 500 mapping, and a custom to_response override attaching WWW-Authenticate.

Verification:
- cargo check --workspace → clean.
- cargo test -p joltr-core --test error → 4 passed.
- cargo test -p joltr-core --test endpoint_integration --test inventory_registration → all passed (sanity sweep on the existing surface most likely to be affected by the new lib.rs re-exports).

Key decisions:
- Chose default-method `to_response(&self) -> Response<ErrorBody>` over a blanket `From<E: JoltRError> for axum::response::Response`. Rationale: a blanket impl risks coherence conflicts with axum's own IntoResponse blanket impls and with the existing `From<Response<T>> for axum::response::Response` bridge, and `to_response()` composes cleanly with that bridge anyway. The endpoint macro (PRD #10) can emit `Err(e) => e.to_response().into()` without further glue.
- `ErrorBody` implements `JsonBody` so the existing `Response<T>` → `axum::response::Response` JSON path is reused — no new serialization code.
- Trait method takes `&self` (not `self`) so a single error value can be inspected by error_handler hooks before conversion, and so `to_response` can be called from `Display`-only matches without consuming the error.
- Did not require `std::error::Error` / `Display` on the trait — keeps the trait usable for enum-only error types without forcing a `thiserror` dep at the joltr-core layer.

Files changed:
- joltr-core/src/error.rs (new)
- joltr-core/src/lib.rs (mod + re-export)
- joltr-core/tests/error.rs (new)
- prd.json (item 9 → passes: true)

Next notes:
- PRD #10 (default error handler in `#[endpoint]` macro) is unblocked. The macro should detect `Result<Response<T>, E>` return types and emit `match` arms that call `err.to_response()` and forward via the existing IntoResponse bridge.
- Optional follow-up: provide a `#[derive(JoltRError)]` (joltr-macros) for enum errors with `#[joltr_error(status = 404, message = "...")]` per-variant attrs. Not in scope here.


2026-05-11 PRD #10 — Default error handler in `#[endpoint]` macro

Completed:
- Extended joltr-macros/src/endpoint.rs:
  - `DiscoveredMethod` carries a new `error_handler: Option<syn::Path>` field captured from a per-method `#[error_handler(fn_path)]` magic-marker attribute.
  - `parse_error_handler_attr` recognizes a bare `error_handler` ident (path-style `#[crate::error_handler(...)]` is intentionally not matched), parses the inner tokens as a `syn::Path`, and surfaces a span-pointed diagnostic when the args are not a function path (e.g. a string literal).
  - `scan_methods` rejects duplicate `#[error_handler]` attrs on the same method.
  - New `ReturnKind { Response, ResultResponse }` returned by `validate_signature` / `validate_return_type`; `validate_methods` uses it to reject `#[error_handler]` on bare `Response<T>` handlers (unreachable glue).
  - `strip_verb_attrs` extended to also strip `#[error_handler(...)]` so the re-emitted impl block does not surface an unknown attribute to rustc.
  - `generate_handler_wrappers` delegates per-method body emission to `generate_wrapper_dispatch`, which:
    - For `Response<T>`: keeps the JOLTR-RS-043 single-line `IntoResponse::into_response(__result)` shape unchanged (no branch).
    - For `Result<Response<T>, E>` with no `#[error_handler]`: emits `match __result { Ok(__ok) => IntoResponse::into_response(__ok), Err(__err) => IntoResponse::into_response(<_ as ::joltr_core::JoltRError>::to_response(&__err)) }`.
    - For `Result<Response<T>, E>` with `#[error_handler(my_fn)]`: emits the same match with `my_fn(__err)` (owned, matches Rust's `match` arm ownership) in the Err arm — no implicit `E: JoltRError` bound.
- Added 9 new unit tests in joltr-macros/src/endpoint.rs (token-shape regression set):
  - `scan_methods` captures simple and path-style `error_handler` args, rejects duplicates and string-literal args.
  - `validate_methods` rejects `#[error_handler]` on `Response<T>` returns and accepts it on `Result<Response<T>, E>` returns.
  - `strip_verb_attrs` also removes `#[error_handler(...)]`.
  - `generate_handler_wrappers` emits match arms + `JoltRError::to_response(&__err)` by default, swaps to `my_handler(__err)` when the attribute is set, and leaves `Response<T>` wrappers branchless.
- Added joltr-core/tests/endpoint_error.rs with 3 end-to-end integration tests through `JoltRServer::into_router()`:
  - `default_err_branch_returns_json_error_body` — `Err(NotFound)` → 404 JSON `{ error, status: 404 }`.
  - `default_ok_branch_still_returns_user_body` — regression: `Ok(Response<Greeting>)` still serializes the user body as JSON 200.
  - `custom_error_handler_replaces_default_bridge` — `#[error_handler(map_custom_err)]` on a non-`JoltRError` error returns 418 with the custom body.

Verification:
- cargo check --workspace → clean.
- cargo test -p joltr-macros --lib → 249 passed (up from 240; 9 new PRD #10 unit tests).
- cargo test -p joltr-core --test endpoint_error → 3 passed.
- cargo test -p joltr-core --test endpoint_integration --test error --test inventory_registration → 9 passed (regression sweep on the surface most likely affected by the wrapper-body refactor).

Key decisions:
- Custom error handler fn takes the error by value (`fn(E) -> impl IntoResponse`), not by reference. Rationale: a `match __result { Err(__err) => fn(__err) }` arm owns the binding, so passing by value gives the user fn full control (move semantics, destructuring) without the macro needing to introduce a temporary or a borrow. Users who want `fn(&E)` can write a one-line shim.
- Re-derive `ReturnKind` at codegen time inside `generate_wrapper_dispatch` instead of threading it through `DiscoveredMethod`. Rationale: the data is cheap to re-derive from `sig.output` and keeping the struct shallow avoids a parallel-state hazard if a future PRD changes the validation pass without updating the codegen consumers.
- Required `JoltRError` bound is implicit — the macro never spells `where E: JoltRError`; rustc surfaces a "trait bound not satisfied" diagnostic at the wrapper site if the user's error type doesn't implement the trait. Mirrors the JOLTR-RS-043 decision to let rustc enforce `Self: Default` rather than have the macro emit an explicit `where` clause.
- `#[error_handler(...)]` is stripped from the re-emitted impl rather than registered as its own proc-macro. Mirrors the verb-marker (`#[get]`, `#[post]`, ...) strategy from 039 — keeps the magic surface narrow to inside `#[endpoint]`.
- Wrapper body for bare `Response<T>` is unchanged (no `match`, no `JoltRError` reference). This pins that the PRD #10 changes pay nothing at the codegen level for handlers that don't have an `Err` branch.

Files changed:
- joltr-macros/src/endpoint.rs (DiscoveredMethod field, attr parser, validation, strip, wrapper-body split, +9 unit tests)
- joltr-core/tests/endpoint_error.rs (new, 3 integration tests)
- prd.json (item 10 → passes: true)

Next notes:
- PRD #9's optional follow-up (`#[derive(JoltRError)]` for enum errors with per-variant `#[joltr_error(status = N, message = "...")]`) remains the natural next polish — would let users skip the manual `match` in `status()`/`message()` and would compose cleanly with the macro-emitted default Err branch landed here.
- The custom-handler path lets the user return any `impl IntoResponse`. If a future PRD wants a stricter contract (e.g. always `Response<ErrorBody>` so error responses are uniformly observable), we can tighten by replacing `IntoResponse::into_response(my_fn(__err))` with an explicit cast — backwards-compatible for users who already return `Response<ErrorBody>`.


2026-05-11 PRD #11 — TsExport: joltr-types binary crate (scaffold)

Completed:
- Added `joltr-types` as a new workspace member (root Cargo.toml `members` list).
- `joltr-types/Cargo.toml` — `[[bin]] name = "joltr-types", path = "src/main.rs"`. Single runtime dep on `inventory` (workspace). No lib target yet; that arrives with PRD #12.
- `joltr-types/src/main.rs` — end-to-end pipeline:
  - `pub struct TsTypeDef { name, body }` placeholder + `inventory::collect!(TsTypeDef)` collection point.
  - `HEADER` const matches the legacy Zig generator's `// === DO NOT MODIFY ===` marker so editors/tooling that key off that string keep working across the port.
  - `render()` walks `inventory::iter::<TsTypeDef>` in insertion order, prefixed by `HEADER`.
  - `resolve_out_path()` honours `JOLTR_TYPES_OUT` first (test sandbox), otherwise writes to `<workspace_root>/types.d.ts` derived from `env!("CARGO_MANIFEST_DIR").parent()`.
  - `main()` writes the rendered string and logs the byte count + destination.
- `joltr-types/tests/ts_export_integration.rs` — 2 integration tests that spawn `CARGO_BIN_EXE_joltr-types` with `JOLTR_TYPES_OUT` redirected into `target/joltr-types-test-<nanos>-<suffix>.d.ts`:
  - `binary_writes_header_to_redirected_output` — exit 0, file contents start with the canonical header.
  - `binary_emits_one_entry_per_inventory_submission` — pins the pre-#12 invariant that the body is exactly the header (no submissions yet); this test will fail loudly when PRD #12 wires `#[derive(TsExport)]` into `inventory::submit!`, surfacing the integration moment.

Verification:
- cargo check --workspace → clean (only new `joltr-types` checked).
- cargo build --workspace → clean (binary + lib targets all link).
- cargo test -p joltr-types → 2 passed (both integration tests).
- cargo test -p joltr-core --test inventory_registration → 9 passed (regression sweep: the new crate adds a second `inventory::collect!` site in the workspace; this confirms the existing endpoint registry collection is unaffected).
- Smoke run: `JOLTR_TYPES_OUT=/tmp/... cargo run -q -p joltr-types` wrote 92 bytes (header only) — pipeline functional.

Key decisions:
- Output path is overridable via `JOLTR_TYPES_OUT` env var. Rationale: a `cargo test` of the binary would otherwise clobber the developer's workspace-level `types.d.ts` (which is `.gitignore`d but still locally meaningful). The env override keeps the integration test hermetic and discoverable on failure (artifacts land in `target/`, not auto-deleted on assertion failure so they can be inspected).
- `TsTypeDef` placeholder is `name + body: &'static str` rather than the richer `kind + fields + generics + docs` shape PRD #12 specifies. Rationale: PRD #11's file list is only `Cargo.toml + src/main.rs` (no `lib.rs`); putting a richer struct in `main.rs` would mean #12 has to delete it and recreate it in `lib.rs`. Keeping the placeholder minimal makes #12's move feel like a refactor rather than a rewrite, and the rendering surface in `render()` is small enough to swap in one edit.
- Binary-only crate (no `[lib]` target yet) — `inventory::submit!` from external crates can't target a symbol that's binary-private, so PRD #12 will add `joltr-types/src/lib.rs` and re-home `TsTypeDef` + `inventory::collect!` there. Calling this out so #12 doesn't get surprised by having to add a `[lib]` target alongside touching `lib.rs`.
- Workspace root resolved via `env!("CARGO_MANIFEST_DIR").parent()` instead of walking up looking for a `Cargo.toml` with `[workspace]`. Rationale: `joltr-types` lives one level deep in the workspace by convention (mirrors `joltr-core`, `joltr-macros`, ...); a parent-of-manifest-dir hop is deterministic and cheap, and the layout move would itself be a workspace-wide refactor that would change this anyway.
- The second integration test (`binary_emits_one_entry_per_inventory_submission`) is intentionally a tripwire for PRD #12. When #12 adds the first `inventory::submit!` from the `#[derive(TsExport)]` proc-macro, this test will fail — that's the signal to update the assertion alongside #12's macro changes, which keeps the inventory plumbing visibly tested rather than silently empty.

Next notes:
