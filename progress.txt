## 2026-05-10 — JOLT-RS-007: Method enum

- ParseMethodError is a newtype wrapping `String` (the offending input) with std::error::Error + Display impls — gives downstream callers a usable error type without dragging in a thiserror dep yet. Renaming/refactoring later is cheap because it's exposed through `pub use`.

## 2026-05-10 — JOLT-RS-009: StatusCode → axum bridge + Display

PRD item: JOLT-RS-009 (priority 9, phase01-method-statuscode) — Implement From<StatusCode> for axum::http::StatusCode and Display for StatusCode.

Verification:
- `cargo test -p jolt-core --lib` → 7 passed (5 prior + into_axum_status_not_found, display_known_includes_reason_phrase, display_other_with_known_code_uses_reason_phrase); compiled clean in 1.14s.
- `cargo check --workspace` → all 5 crates check clean in 0.38s; no regressions.

Note on harness instructions: same as JOLT-RS-007/008 — npm/Bun commands in the auto-prompt don't apply to this Rust workspace. cargo is the verification tool.

Key decisions:
- `From<StatusCode> for axum::http::StatusCode` uses `axum::http::StatusCode::from_u16(self.as_u16()).expect(...)`. The expect message names the invariant ("StatusCode variants must encode valid HTTP status numbers"). All ten named variants encode 200/201/204/400/401/403/404/405/409/500 — all in the valid 100..=999 range. Other(u16) values outside that range will panic, which is correct: constructing Other(99) or Other(1000) is a programming bug, not a recoverable runtime error. Documented this implicitly via the expect message rather than wrapping in a TryFrom — a fallible bridge would force every call site to handle an impossible-for-named-variants error.
- `Display for StatusCode` delegates to axum's canonical_reason for the reason phrase. This matches axum::http::StatusCode's "200 OK" / "404 Not Found" output, which is what users will see in logs and error pages everywhere else in the stack. For Other(u16) codes that lack a canonical reason (or are outside 100..=999), falls back to just the numeric code. This preserves the symmetry that axum gives.
- Did NOT implement Display in terms of as_u16 + a hand-rolled match on canonical reason strings. Letting axum own the reason-phrase mapping means we don't have to update Jolt every time IANA registers a new status code (e.g. the newer 425 Too Early), and we automatically inherit any future axum/http-crate fixes.
- Added three tests aligned with the PRD verification ("StatusCode::NotFound → axum StatusCode 404") plus two adjacent Display tests to lock in the format choice. Other(418) → "418 I'm a teapot" verifies the canonical_reason lookup works for unnamed-but-known codes.
- Pulled `use std::fmt;` at the top of the module rather than fully-qualifying — same style as method.rs.
