## 2026-05-10 — JOLT-RS-033: tower::Service for Router (opens phase07)

PRD item: JOLT-RS-033 (priority 33, phase07-router-tower-service) — Implement `tower::Service` for `Router`: wraps an `axum::Router` and injects `RequestExt` into every incoming request before dispatch.

Why picked over the strict-priority next items (JOLT-RS-027 port-binding-conflict polish; JOLT-RS-032 endpoint-registry test bundle): per the auto-prompt's prioritization rule ("architectural decisions and core abstractions" first), JOLT-RS-033 is the load-bearing entry point for phase07. The whole middleware layering story (CORS in phase09, body parsing in phase10, query parsing in phase11, tracing in phase12, JWT auth in phase13, WS auth in phase14) reads `Arc<RequestExt>` out of request extensions and toggles its `finished` latch. Without 033's injection at the outermost service layer, none of those middlewares can exist as standalone tower layers — they'd each need to lazy-init the latch defensively, which fragments the contract. Landing 033 first means 034/035/036/037 (and every later middleware PRD) can assume the extension is always present and read-only-shaped (the latch itself is `AtomicBool`, hence `Arc` rather than `Box`). JOLT-RS-027 stays a polish task with no upstream dependencies; JOLT-RS-032 is a test bundle that's blocked on small but unresolved design questions (duplicate-path detection error shape, path-param Request field) that should land alongside an actual consumer rather than ahead of one.

- `cargo test -p jolt-core --lib -- tests::router` → 3/3 passed (Service impl compiles + 2 behavioral tests covering the injection contract and inner-router transparency), 85 filtered out, 0.00s.
- `cargo test -p jolt-core --lib` → 88 passed total (up from 85); 0 failed; 0 ignored.
- `cargo check --workspace` → clean in 0.38s; no regressions in any sibling crate.

Note on harness instructions: same as JOLT-RS-007..031 — auto-prompt's `npm run type-check` / Bun-Hono server commands don't apply to this Rust workspace; cargo is the verification tool.

Key decisions:
- New file `jolt-core/src/router.rs` (one-type-per-file, matching the existing convention: `cookie.rs`, `endpoint.rs`, `endpoint_registry.rs`, `method.rs`, `request.rs`, `request_ext.rs`, `response.rs`, `server.rs`, `status.rs`). Re-exported as `pub use router::Router` from `lib.rs` so callers write `jolt_core::Router` (matches the PRD-036 `Router::new(EndpointRegistry)` shape that lands later).
- **Storage shape: `inner: axum::Router` (one field).** Keeping the wrapper a thin pass-through is the minimum the PRD step asks for. The handler-dispatch + finished-flag short-circuit are explicitly JOLT-RS-034/035 territory; trying to combine them with 033 here would conflate three different changes and make the eventual 035 test set harder to scope. The doc-comment names which sibling PRD owns each follow-up so the next iteration has a checklist.
- **Injection target: `Arc<RequestExt>`, not `RequestExt` itself.** The latch is `AtomicBool`-backed, so cloning is the correct sharing primitive when downstream middlewares spawn parallel work or need to capture the latch into a future. Box would force one-owner semantics that the existing `RequestExt` shape (`AtomicBool` for `finished`) explicitly avoids. Stored under the typed extension key `Arc<RequestExt>` so subsequent middlewares fetch with `req.extensions().get::<Arc<RequestExt>>()`. This is the canonical contract every phase09+ middleware will read.
- **Overwrite-on-call semantics for the extension.** `req.extensions_mut().insert(Arc::new(RequestExt::new()))` unconditionally replaces any previously inserted `Arc<RequestExt>`. Pinned by the `service_call_overwrites_caller_supplied_request_ext` test: a caller that pre-populated a finished latch (e.g. some upstream proxy/test fixture) MUST NOT leak that flag into the handler. Otherwise JOLT-RS-035's short-circuit contract is poisoned at the entry. This is one of the non-obvious correctness invariants of phase07; documenting it as a test now means a future "performance" optimization that skips the insert when one already exists trips the test.
- **Constructor: `Router::from_axum(inner: axum::Router)` only.** Did NOT add `Router::new()`. PRD-036 reserves `pub fn new(registry: EndpointRegistry) -> Self` as the primary public constructor. A no-arg `new()` would shadow that. `from_axum` is a deliberately lower-level entry that stays useful for tests + future composition helpers (mounting an axum::Router built externally, e.g. for serving static assets or third-party axum routers). Document the JOLT-RS-036 boundary in the rustdoc so the constructor proliferation is intentional, not accidental.
- **`#[derive(Clone)]` on Router.** `axum::Router` is `Clone` (cheap — internally a `MakeService` that clones cheaply), and tower::Service callers commonly clone services to call from multiple tasks. Skipping Clone would force every test that wants to call the service twice to rebuild it. Cost is zero; benefit is real.
- **Service impl shape: `Service<axum::extract::Request>`** (axum's default `Request<Body>`), `type Response = axum::response::Response`, `type Error = Infallible`, `type Future = Pin<Box<dyn Future<Output = Result<Response, Infallible>> + Send>>`. Matches axum::Router<()>'s own Service signature exactly so the wrapper can be substituted at any axum::serve call site. Boxed future because the call body is async (`inner.call(req).await`); Box<dyn Future> avoids leaking RouteFuture<Infallible> from axum's internals into Router's public API and keeps the type stable across axum upgrades. Tradeoff: per-request heap allocation for the boxed future. Acceptable here — 034's full registry-driven dispatch will need an async path anyway, and the alternative (a manual `enum Future { ... }`) is significant complexity for a still-evolving service shape.
- **`poll_ready` forwards to inner.** Uses qualified path `<axum::Router as Service<AxumRequest>>::poll_ready(&mut self.inner, cx)` to disambiguate; axum::Router has multiple Service impls in scope through Tower's blanket impls and the explicit qualification spells out exactly which one we mean.
- **`call`'s clone-and-move pattern** mirrors the canonical tower wrapper recipe: `let mut inner = self.inner.clone(); Box::pin(async move { inner.call(req).await })`. The clone is cheap (axum::Router uses Arc internally for its routes); the move into the async block avoids holding `self`'s mutable reference across the `.await`, which the borrow checker otherwise rejects.
- **Did NOT add `Default for Router`**. Same rationale as JOLT-RS-023's `JoltServer` decision: no clippy gate yet, and `from_axum` is the construction story until 036 lands `new(registry)`. Once 036 is in, `Default` could derive cleanly via `Default::default() -> Self::new(EndpointRegistry::default())`.
- **Three tests under new `mod router` (inserted between `mod endpoint_registry` and `mod server` to follow the dependency order in lib.rs):**
  1. `service_call_injects_request_ext_into_extensions` — happy-path injection. Uses an in-process probe handler that inspects `req.extensions().get::<Arc<RequestExt>>()` and returns 200 / 500 based on presence. Asserts the freshly-injected latch defaults to not-finished, locking in `RequestExt::new()`'s contract from the consumer side.
  2. `service_call_overwrites_caller_supplied_request_ext` — the non-obvious invariant. Pre-inserts a finished `Arc<RequestExt>` on the request, then verifies the handler sees a not-finished latch. This is the test that catches a future "skip insert if already present" optimization.
  3. `service_call_forwards_to_inner_axum_router_for_unmatched_paths` — transparency on the routing dimension. Empty inner Router → 404 still flows through the wrapper. Pins that 033 hasn't accidentally introduced a catch-all that swallows axum's default not-found behavior.
- **`tower::ServiceExt::oneshot`** for test invocation — same fixture pattern as `mod endpoint_registry::build_router`. The workspace `tower` dep already ships with `features = ["util"]` (added in JOLT-RS-031), so no new dep work.
- **No new dependencies.** All imports are already in jolt-core's tree: `axum::{body::Body, extract::Request, http::StatusCode, routing::get}`, `tower::{Service, ServiceExt}`, `std::{sync::Arc, convert::Infallible, future::Future, pin::Pin, task::{Context, Poll}}`.
