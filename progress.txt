## 2026-05-10 — JOLT-RS-007: Method enum

PRD item: JOLT-RS-007 (priority 7, phase01-method-statuscode) — Define Method enum {Get, Post, Put, Patch, Delete, Options, Head} with Display + FromStr in jolt-core. Opens phase01.

Verification:
- `cargo test -p jolt-core --lib` → 2 passed (from_str_get_is_ok, from_str_invalid_is_err); compiled clean in 6.50s.
- `cargo check --workspace` → all 5 crates check clean in 8.60s; no regressions in jolt-macros/jolt-db/jolt-utils/jolt-templates.

Note on harness instructions: the auto-generated task prompt referenced `npm run type-check` and a Bun/Hono server, which are leftovers from a prior Node/TS incarnation of this repo. The current repo is a pure Rust workspace (no package.json, no server/ dir), so verification used `cargo test` + `cargo check --workspace` instead. Future iterations should keep using cargo until/unless package.json reappears.

Key decisions:
- FromStr is case-sensitive against canonical uppercase names (RFC 9110 §9). The PRD verification only tests "GET" → Ok(Get); broader case handling (lowercase/mixed) is not in scope and would silently swallow malformed wire input. If callers need case-insensitive parsing later, they should `s.to_ascii_uppercase()` at the call site rather than baking it into the framework.
- Variant order matches the PRD spec exactly (Get, Post, Put, Patch, Delete, Options, Head) — preserving the spec ordering helps when JOLT-RS-010 writes round-trip tests.
- Added a `Method::as_str() -> &'static str` inherent helper that Display delegates to. This avoids allocating a String just to format the wire name and gives router code a zero-cost path to the canonical bytes. Not in the PRD step list but trivial and load-bearing for future routing work.
- ParseMethodError is a newtype wrapping `String` (the offending input) with std::error::Error + Display impls — gives downstream callers a usable error type without dragging in a thiserror dep yet. Renaming/refactoring later is cheap because it's exposed through `pub use`.
- Re-exported Method + ParseMethodError at the crate root via `pub use method::{Method, ParseMethodError}` — keeps the public API ergonomic (`jolt_core::Method`) while the module structure stays intact for future additions.
- Did NOT yet implement `From<Method> for axum::http::Method` or `serde::{Serialize, Deserialize}` — those are separate concerns and not in JOLT-RS-007's scope. JOLT-RS-009 handles the StatusCode↔axum bridge and may inspire a parallel Method bridge in a later PRD item.

Files changed:
- jolt-core/src/method.rs (new) — Method enum, Display, FromStr, ParseMethodError, 2 unit tests.
- jolt-core/src/lib.rs — added `pub mod method;` and `pub use method::{Method, ParseMethodError};`.
