## 2026-05-10 — JOLT-RS-012: Request::header(name) accessor

PRD item: JOLT-RS-012 (priority 12, phase02-request-type) — Implement Request::header(name: &str) → Option<&str> with case-insensitive lookup.

Verification:
- `cargo test -p jolt-core --lib` → 23 passed (20 prior + tests::request::header_lookup_is_case_insensitive, tests::request::header_returns_none_for_missing_name, tests::request::header_returns_none_for_non_visible_ascii_value); compiled clean in 1.09s.

Note on harness instructions: same as JOLT-RS-007..011 — auto-prompt's `npm run type-check` / Bun-Hono server commands don't apply to this Rust workspace. cargo is the verification tool.

Key decisions:
- Implementation is one line: `self.headers.get(name).and_then(|v| v.to_str().ok())`. axum::http::HeaderMap::get accepts `&str` (via the AsHeaderName impl that constructs a HeaderName, which lower-cases the input on the way in), so case-insensitive lookup falls out of the underlying type — no manual normalization required. Extra abstractions would just hide what the HeaderMap already does.
- Return type is `Option<&str>`, not `Option<&HeaderValue>` or `Result<Option<&str>, _>`. The PRD spec says `Option<&str>`, and `HeaderValue::to_str()` only succeeds for visible-ASCII values (HTTP !..=~ plus space/tab). For binary header values (rare in practice — non-ASCII bytes only show up in things like custom protocol extensions or malformed proxies) `to_str` returns Err, which we collapse to None. This matches the principle "if you can't render it as &str, the caller asking for &str should treat it as absent." Documented the boundary in a one-line doc comment so a future reader doesn't read the elision as "&str-coerce always succeeds."
- Added two negative tests beyond the PRD verification: missing-name returns None (locks in the basic absence path), and binary-value returns None (locks in the to_str() fallback). These are not scope creep — they are the only two ways `header()` can return None, and without them the function's contract isn't fully test-pinned. Total cost: ~10 lines of test for two behaviors that would otherwise be implicit.
- Refactored the request test module to use a `fn empty_request() -> Request` helper. Three new tests all needed a baseline Request and the existing `struct_literal_construction_reaches_every_field` test verifies struct-literal construction independently — extracting a helper for the rest avoids 30 lines of repeated boilerplate. Helper lives in the test module only; not part of the public API. If JOLT-RS-013/014 want it too, it's available.
- Did NOT add a `headers_mut()` accessor or expose any other HeaderMap-shaped API. The Request struct's `headers` field is already pub (per JOLT-RS-011's decision), so callers can mutate the underlying map directly. Adding wrapper methods now would be premature.

- Removed the JOLT-RS-011 doc comment in request.rs that read "Field-only definition for JOLT-RS-011. Accessors (header, query, json, cookie, has_finished) land in JOLT-RS-012..014." That comment was useful as a forward-pointer when 011 was the latest item; now that 012 has landed and the `header` accessor is here, the comment is half-obsolete and would rot further as 013/014 land.
