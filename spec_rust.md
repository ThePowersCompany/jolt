# Plan: Port Jolt (Zig) to Rust

## Foundation Choice

| Zig Component | Rust Replacement | Rationale |
|---|---|---|
| facil.io (C) | **Axum** (tokio + hyper + tower) | Async HTTP, composable tower middleware, WS via `axum::extract::ws`, no C FFI |
| comptime reflection | **Proc macros** (`#[endpoint]`, `#[derive(AutoMiddleware)]`) | Same compile-time codegen, cleaner error messages |
| pg.zig | **sqlx** | Async, compile-time query checking, built-in `sqlx::migrate!` |
| fiobj_mustache | **handlebars** | Mustache-compatible, Rust-native |
| fio_run_every | **tokio::time::interval** | Standard async scheduling |
| HTTP/sendfile/TLS | **tower-http** (fs, cors, trace) + **rustls** + **axum-server** | Production-grade, no C deps |
| JWT (custom HS256) | **jsonwebtoken** | Well-audited, full JWS support |

## Phase Breakdown

### Phase 1 ‚Äî Foundation: Server, Router, Request/Response

- `jolt-core`: `JoltServer` builder (port, threads ‚Üí tokio workers, CORS config), endpoint registry with `startsWith` path sorting, `Request`/`Response<T>` types, `StatusCode` enum, `Method` enum, JSON serialization via serde
- `jolt-macros`: `#[endpoint(path)]` attribute macro that discovers `#[get]`/`#[post]`/... methods via proc macro on impl blocks
- Tower-compatible `Router` that sorts routes by path-length-descending (matching Jolt's `Endpoint.Listener` behavior)

### Phase 2 ‚Äî Middleware Pipeline

- `#[derive(AutoMiddleware)]` ‚Äî inspects struct fields, injects `req` reference, auto-applies body parsing (if `body: T` field exists), query parsing (if `query_params` field exists), CORS (if `#[cors]` attribute present), then user-defined middleware fns
- Port `cors.zig`, `parse-body.zig` (JSON + string via serde), `parse-query-params.zig` (typed param extraction: int/float/bool/string/enum/array), `log.zig`, `auth.zig` (JWT Bearer + WS subprotocol)
- Semaphore-based early termination (`req.is_finished()` ‚Üí tower's `Response` shortcut)

### Phase 3 ‚Äî Database & Templates

- `jolt-db`: sqlx connection pool, `query_as` macros, `LISTEN/NOTIFY` via tokio-postgres notification stream
- `jolt-db/migrations`: File-based SQL migrations with SHA-256 checksums (port `db-migrations.zig`), auto-create `_migrations` table, rollback detection, new migration scaffolding CLI task
- `jolt-templates`: Handlebars rendering with Rust struct ‚Üí template data conversion (port `mustache.zig`)
- `jolt-macros`: `#[derive(PatchQuery)]` ‚Äî proc macro that generates dynamic SQL UPDATE/UPSERT from struct fields with `Optional<T>` handling (port `patch.zig`)

### Phase 4 ‚Äî Real-time: WebSockets, Pub/Sub, SSE, Tasks

- `jolt-core/websocket`: `WebSocketHandler` trait with `on_open`/`on_message`/`on_close`/`on_ready`/`on_shutdown` callbacks, `ws!` macro for subprotocol auth
- `jolt-core/pubsub`: Channel-based publish/subscribe using `tokio::sync::broadcast` behind an abstraction that mirrors facil.io's `fio_publish`/`websocket_subscribe`
- `jolt-core/sse`: SSE support via `axum::response::Sse` ‚Äî port the SSE open/close/ready/shutdown callback pattern
- `jolt-core/task`: Background task scheduler using `tokio::time::interval`, arena-wrapped tasks, retry logic (port `task.zig`)

### Phase 5 ‚Äî Utilities & TypeScript Typegen

- `jolt-utils`: JWT (HS256/384/512 via `jsonwebtoken`), PBKDF2 password hashing, UUID v4/v7, RFC3339 datetime parsing, `Json<T>`/`JsonArray<T>`/`Optional<T>` wrapper types with serde integration, email utils, MIME table
- `jolt-macros`: `#[derive(TsExport)]` ‚Äî proc macro that generates `types.d.ts` from Rust endpoint structs, supporting unions, enums, optionally-tagged unions, `JsonArray`, `Optional` (port `typegen.zig`)
- `types.d.ts` output parity with the existing Zig typegen

### Phase 6 ‚Äî TLS, File Serving, Polish

- TLS via `axum-server` with `rustls` (SNI, cert loading, trust lists ‚Äî port `tls.zig`)
- File serving with gzip, range requests, cache headers via `tower-http::services::ServeFile`
- Cookie parsing/setting (port cookie logic from `request.zig`)
- Redirect helper
- Error handling: custom error handler per endpoint, typed error propagation
- Full example application with endpoints, WS chat, background task, DB migrations

## Key Design Decisions

1. **Middleware short-circuiting**: Tower middleware layers with a `RequestExt` extension that carries a `finished: bool` flag. Each layer checks this before proceeding ‚Äî matches Jolt's `req.markAsFinished()`.

2. **`Optional<T>`**: A three-state enum ‚Äî `Some(T)`, `Null` (explicitly null in JSON), `NotProvided` (field absent). Maps to Jolt's tri-state `Optional` type, essential for PATCH/UPSERT semantics.

3. **WebSocket auth**: Jolt extracts JWT from the `Sec-WebSocket-Protocol` header. Same approach in Rust ‚Äî the WS upgrade handler validates the token before upgrading.

4. **Pub/sub**: facil.io's `fio_publish`/`websocket_subscribe` maps to `tokio::sync::broadcast` channels keyed by channel name in a `DashMap<String, broadcast::Sender>`.

5. **TypeScript typegen**: Proc macro walks the AST of endpoint return types at compile time, writing `types.d.ts` as a side effect (opt-in via a `jolt-types` binary, similar to how `zig build types` works).
